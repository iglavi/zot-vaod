# ⚖️ גילוי נאות — חיפוש הליכים משפטיים

אתר לחיפוש במאגר פסקי דין והחלטות מתוך אתר הרשות השופטת.

לאפליקציה שני מנועי חיפוש:

| | |
|---|---|
| **🔍 חיפוש רגיל** | לפי שם צד, שם שופט/ת, מספר תיק, בית משפט, סוג הליך, טווח תאריכים וטקסט חופשי בגוף פסק הדין. |
| **✨ חיפוש חכם (AI)** | שאלות בשפה חופשית (למשל: "אילו פסקי דין מהשנה האחרונה עסקו בסכסוכי עבודה?") שמקבלות תשובה מנומקת עם הפניות למספרי התיקים. מבוסס על מודל Claude של Anthropic. |

---

## התקנה (פעם אחת)

1. התקינו Python 3.10 ומעלה.
2. בשורת הפקודה, בתוך תיקיית הפרויקט, הריצו:

   ```
   pip install -r requirements.txt
   ```

## הכנת הנתונים

האפליקציה עובדת עם שני מקורות:

1. **`data/metadata.csv`** — קובץ המטא-דאטה של כל ההחלטות (עמודות: תאריך, בית משפט,
   הליך, סוג תיק, סוג עניין, מספר תיק, שם תיק, סוג החלטה, אופי החלטה, `file_path`).
2. **תיקיית פסקי הדין** — קבצי ה-`.docx` שירדו מנט-המשפט. ברירת המחדל היא התיקייה
   `documents/`, אך אפשר להצביע על התיקייה של סקריפט ההורדה (כולל תת-תיקיות לפי תאריך)
   באמצעות משתנה הסביבה `ZOT_DOCS` — ראו למטה.

> החיבור בין שורה ב-CSV לקובץ פסק הדין נעשה לפי שם הקובץ (מתוך העמודה `file_path`).
> החיפוש הרגיל עובד גם על החלטות שאין להן קובץ `.docx` מקומי (על סמך המטא-דאטה בלבד);
> חיפוש הטקסט המלא והחיפוש החכם עובדים על ההחלטות שיש להן קובץ מלא.

## בניית האינדקס

לאחר שהנתונים במקום, בנו את אינדקס החיפוש (מסד נתונים מקומי מסוג SQLite עם חיפוש
טקסט מלא). הריצו פעם אחת, ובכל פעם שמתווספים פסקי דין חדשים:

```
python -m zot.ingest
```

(אפשר גם ללחוץ על הכפתור **"בנה אינדקס עכשיו"** בתוך האפליקציה בפעם הראשונה.)

## הפעלה

```
python -m streamlit run app.py
```

הדפדפן ייפתח אוטומטית עם האפליקציה.

## חיפוש חכם (AI) — הגדרת מפתח

מנוע החיפוש החכם משתמש במודל Claude ולכן דורש מפתח API של Anthropic.
הגדירו את המפתח כמשתנה סביבה **לפני** הפעלת האפליקציה:

- **Windows (Command Prompt):**
  ```
  set ANTHROPIC_API_KEY=sk-ant-...
  python -m streamlit run app.py
  ```
- **Mac / Linux:**
  ```
  export ANTHROPIC_API_KEY=sk-ant-...
  python -m streamlit run app.py
  ```

ללא מפתח — החיפוש הרגיל פועל כרגיל, ורק לשונית ה-AI מציגה הודעה מתאימה.

## איך זה עובד (מבנה הפרויקט)

```
zot-vaod/
├── app.py              # ממשק המשתמש (Streamlit) — שתי הלשוניות
├── zot/
│   ├── config.py       # הגדרות ונתיבים
│   ├── extract.py      # חילוץ טקסט + שופט + תאריך החלטה מקבצי docx/txt
│   ├── ingest.py       # בניית האינדקס (SQLite + FTS5) מ-CSV + פסקי דין
│   ├── search.py       # שאילתות חיפוש רגיל וחיפוש טקסט מלא
│   └── ai_search.py    # מנוע ה-RAG: ניתוח שאלה → אחזור → תשובה מנומקת (Claude)
├── data/metadata.csv   # המטא-דאטה
├── documents/          # קבצי פסקי הדין (.docx / .txt)
└── requirements.txt
```

**מנוע החיפוש החכם (RAG)** פועל בשני שלבים: (1) המודל מנתח את השאלה בעברית וממיר
אותה למילות חיפוש ולטווח תאריכים (כולל פענוח ביטויים כמו "בשנה האחרונה"); (2) המערכת
מאתרת את פסקי הדין הרלוונטיים ביותר מתוך האינדקס ומעבירה אותם למודל, שמנסח תשובה
**רק על סמך פסקי הדין שסופקו**, עם הפניות למספרי התיקים. כך התשובות מבוססות על המאגר
האמיתי ולא על "ידע כללי" של המודל.

## פרטיות

כל האינדקס והחיפוש רצים מקומית על המחשב. בחיפוש החכם בלבד, טקסט פסקי הדין הרלוונטיים
לשאלה נשלח ל-API של Anthropic לצורך ניסוח התשובה.

---

## ארכיטקטורת האינדקס: Turso + R2 (מומש)

**הבעיה שנפתרה:** קובץ האינדקס (`data/index.db`) התארח ב-Cloudflare R2 והיה מורד
בשלמותו (כ-5GB, צפוי לגדול פי 3) לכל מופע-שרת שקם מחדש ב-Streamlit Cloud (הפעלה
קרה, פריסה מחדש, או פקיעת מטמון של שעה) — נמדד בפועל בכ-60+ שניות בלי שום חיווי
חזותי. מדידה בפועל הראתה שטקסט פסקי הדין (`full_text`) הוא רק כ-1.9GB מתוך הקובץ
— אינדקס החיפוש המלא (FTS5) עצמו שקל כ-2.7GB נוספים, כך שהוצאת הטקסט הגולמי בלבד
הייתה פותרת רק כ-40% מהבעיה.

**הפתרון:** המטא-דאטה ואינדקס ה-FTS5 (בלי טקסט פסקי הדין המלא) עברו ל-**Turso**
(מסד נתונים מרוחק, תואם-SQLite/FTS5, שירות מנוהל לגמרי) — האתר שולח שאילתות
ישירות דרך הרשת ולא מוריד שום קובץ בהפעלה (`app.py`: `ensure_index_ui` מדלגת על
כל מנגנון הסנכרון/בנייה המקומי כש-`TURSO_DATABASE_URL` מוגדר). טקסט פסקי הדין
המלא עצמו נשאר ב-R2, מסמך בודד לכל פסק דין (`zot/storage.py`:
`upload_fulltext`/`fetch_fulltext(s)`, דחוס gzip) ונשלף לפי דרישה בלבד. אחרי
`VACUUM`, הנפח בפועל ב-Turso הוא כ-**1.1GB** (678,397 רשומות) — נוח בתוך המסלול
החינמי (5GB).

**מלכודת ביצועים קריטית שנמצאה ותוקנה:** שאילתת `COUNT(*)` על JOIN בין `verdicts`
ל-`verdicts_fts` (עם `MATCH`) לקחה 20-26 שניות בפועל מול Turso (סריקה מלאה של
כ-1.28M שורות), ללא קשר לסדר ה-JOIN — למרות ש-SQLite מקומי מטפל בזה מיידית.
הסיבה: Turso המנוהל **חוסם `ANALYZE`** (`SQL not allowed statement: ANALYZE`),
כך שתכנון-השאילתה נשאר גרוע לצמיתות בלי הסטטיסטיקות שהיו מתקנות זאת מקומית.
שאילתות עם `LIMIT`+`ORDER BY` (כולל `bm25()`) לא נפגעו כלל (עשרות מ"ש) — הבעיה
ספציפית ל-`COUNT(*)` בלי הגבלה. הפתרון: כל ספירה כוללת (ב-`simple_search`
ו-`count_verdicts`) משתמשת תמיד ב-`id IN (SELECT rowid FROM verdicts_fts WHERE
MATCH ?)` במקום JOIN — מהיר (< 1ms) בכל מקרה, כי הספירה ממילא לא זקוקה ל-bm25.

**מלכודת נוספת:** טבלת FTS5 "חסרת-תוכן" (`content=''`) לא תומכת ב-`DELETE FROM
verdicts_fts WHERE rowid=?` רגיל — חובה `INSERT INTO verdicts_fts(verdicts_fts,
rowid, ...) VALUES ('delete', ...)` עם הערכים הישנים במפורש (ראו
`confidentiality_bot.py`/`zot/turso_sync.py: delete_verdicts`). תוקן גם הצורך
למחוק את אובייקט הטקסט המלא מ-R2 עצמו בכל מחיקת רשומה (`storage.delete_fulltext`)
ולמחוק את הרשומה גם מ-Turso (לא רק מהעותק המקומי) — אחרת תיק חסוי שהוסר מקומית
נשאר גלוי וניתן לחיפוש באתר החי.

**קבצים מרכזיים:**
- `zot/storage.py` — `upload_fulltext(s)`/`fetch_fulltext(s)`/`delete_fulltext`.
- `zot/ingest.py` — בלי עמודת `full_text` (יש `text_length` למיון); `verdicts_fts`
  היא `content=''`; העלאת טקסטי מסמכים חדשים ל-R2 קורית אחרי commit (לא בתוכו).
- `zot/search.py` — חיבור ל-Turso (`libsql`) או SQLite מקומי, שקוף לקוד הקורא.
- `zot/turso_sync.py` — סנכרון יומי מצטבר (HTTP API, לא `libsql` — כי `fetch_daily.py`/
  `ingest_loop.py` רצים על Python מקומי שאין לו wheel בנוי מראש ל-`libsql`) +
  `delete_verdicts` למחיקות חיסיון.
- `_migrate_to_r2_turso.py` — סקריפט ההעברה החד-פעמית שהורץ (678,397 רשומות).
- `zot/ai_search.py`/`zot/config.py` — `AI_MAX_DOCS` הוגדל ל-20, אין יותר גילוי
  "מדגם" בתשובה, ושדה `intent` מנתב שאלות מטא (כמו "מה ההחלטה העדכנית ביותר")
  למיון-לפי-תאריך במקום BM25.
