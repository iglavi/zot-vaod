# ⚖️ גילוי נאות — חיפוש הליכים משפטיים

אתר לחיפוש במאגר פסקי דין והחלטות מתוך אתר הרשות השופטת.

לאפליקציה שני מנועי חיפוש:

| | |
|---|---|
| **🔍 חיפוש רגיל** | לפי שם צד, שם שופט/ת, מספר תיק, בית משפט, סוג הליך, טווח תאריכים וטקסט חופשי בגוף פסק הדין. |
| **✨ חיפוש חכם (AI)** | שאלות בשפה חופשית (למשל: "אילו פסקי דין מהשנה האחרונה עסקו בסכסוכי עבודה?") שמקבלות תשובה מנומקת עם הפניות למספרי התיקים. מבוסס על מודל Claude של Anthropic. |

---

## התקנה (פעם אחת)

1. התקינו Python 3.10 ומעלה.
2. בשורת הפקודה, בתוך תיקיית הפרויקט, הריצו:

   ```
   pip install -r requirements.txt
   ```

## הכנת הנתונים

האפליקציה עובדת עם שני מקורות:

1. **`data/metadata.csv`** — קובץ המטא-דאטה של כל ההחלטות (עמודות: תאריך, בית משפט,
   הליך, סוג תיק, סוג עניין, מספר תיק, שם תיק, סוג החלטה, אופי החלטה, `file_path`).
2. **תיקיית פסקי הדין** — קבצי ה-`.docx` שירדו מנט-המשפט. ברירת המחדל היא התיקייה
   `documents/`, אך אפשר להצביע על התיקייה של סקריפט ההורדה (כולל תת-תיקיות לפי תאריך)
   באמצעות משתנה הסביבה `ZOT_DOCS` — ראו למטה.

> החיבור בין שורה ב-CSV לקובץ פסק הדין נעשה לפי שם הקובץ (מתוך העמודה `file_path`).
> החיפוש הרגיל עובד גם על החלטות שאין להן קובץ `.docx` מקומי (על סמך המטא-דאטה בלבד);
> חיפוש הטקסט המלא והחיפוש החכם עובדים על ההחלטות שיש להן קובץ מלא.

## בניית האינדקס

לאחר שהנתונים במקום, בנו את אינדקס החיפוש (מסד נתונים מקומי מסוג SQLite עם חיפוש
טקסט מלא). הריצו פעם אחת, ובכל פעם שמתווספים פסקי דין חדשים:

```
python -m zot.ingest
```

(אפשר גם ללחוץ על הכפתור **"בנה אינדקס עכשיו"** בתוך האפליקציה בפעם הראשונה.)

## הפעלה

```
python -m streamlit run app.py
```

הדפדפן ייפתח אוטומטית עם האפליקציה.

## חיפוש חכם (AI) — הגדרת מפתח

מנוע החיפוש החכם משתמש במודל Claude ולכן דורש מפתח API של Anthropic.
הגדירו את המפתח כמשתנה סביבה **לפני** הפעלת האפליקציה:

- **Windows (Command Prompt):**
  ```
  set ANTHROPIC_API_KEY=sk-ant-...
  python -m streamlit run app.py
  ```
- **Mac / Linux:**
  ```
  export ANTHROPIC_API_KEY=sk-ant-...
  python -m streamlit run app.py
  ```

ללא מפתח — החיפוש הרגיל פועל כרגיל, ורק לשונית ה-AI מציגה הודעה מתאימה.

## הגדרות מתקדמות (משתני סביבה — לא חובה)

| משתנה | ברירת מחדל | תיאור |
|---|---|---|
| `ANTHROPIC_API_KEY` | — | מפתח ה-API עבור החיפוש החכם. |

## איך זה עובד (מבנה הפרויקט)

```
zot-vaod/
├── app.py              # ממשק המשתמש (Streamlit) — שתי הלשוניות
├── zot/
│   ├── config.py       # הגדרות ונתיבים
│   ├── extract.py      # חילוץ טקסט + שופט + תאריך החלטה מקבצי docx/txt
│   ├── ingest.py       # בניית האינדקס (SQLite + FTS5) מ-CSV + פסקי דין
│   ├── search.py       # שאילתות חיפוש רגיל וחיפוש טקסט מלא
│   └── ai_search.py    # מנוע ה-RAG: ניתוח שאלה → אחזור → תשובה מנומקת (Claude)
├── data/metadata.csv   # המטא-דאטה
├── documents/          # קבצי פסקי הדין (.docx / .txt)
└── requirements.txt
```

**מנוע החיפוש החכם (RAG)** פועל בשני שלבים: (1) המודל מנתח את השאלה בעברית וממיר
אותה למילות חיפוש ולטווח תאריכים (כולל פענוח ביטויים כמו "בשנה האחרונה"); (2) המערכת
מאתרת את פסקי הדין הרלוונטיים ביותר מתוך האינדקס ומעבירה אותם למודל, שמנסח תשובה
**רק על סמך פסקי הדין שסופקו**, עם הפניות למספרי התיקים. כך התשובות מבוססות על המאגר
האמיתי ולא על "ידע כללי" של המודל.

## פרטיות

כל האינדקס והחיפוש רצים מקומית על המחשב. בחיפוש החכם בלבד, טקסט פסקי הדין הרלוונטיים
לשאלה נשלח ל-API של Anthropic לצורך ניסוח התשובה.

## תכנון ארכיטקטורה עתידי (טרם בוצע)

**הבעיה:** קובץ האינדקס (`data/index.db`) מתארח היום ב-Cloudflare R2 ומורד בשלמותו
(כ-5GB, צפוי לגדול פי 3) לכל מופע-שרת שקם מחדש ב-Streamlit Cloud (הפעלה קרה,
פריסה מחדש, או פקיעת מטמון של שעה) — נמדד בפועל בכ-60+ שניות בלי שום חיווי חזותי.
מדידה בפועל הראתה שטקסט פסקי הדין (`full_text`) הוא רק כ-1.9GB מתוך הקובץ — אינדקס
החיפוש המלא (FTS5) עצמו שוקל כ-2.7GB נוספים, כך שהוצאת הטקסט הגולמי בלבד הייתה
פותרת רק כ-40% מהבעיה.

**הכיוון שנבחר:** להעביר את המטא-דאטה ואת אינדקס ה-FTS5 (בלי טקסט פסקי הדין המלא)
ל-**Turso** (מסד נתונים מרוחק, תואם-SQLite/FTS5, שירות מנוהל לגמרי — בלי שרת
עצמאי לתחזק) — כך שהאתר ישלח שאילתות ישירות דרך הרשת ולא יצטרך להוריד קובץ בכלל.
טקסט פסקי הדין המלא עצמו נשאר ב-R2, מסמך בודד לכל פסק דין (אותו דפוס אחסון
שכבר קיים היום לקבצי PDF/Word המקוריים) ונשלף לפי דרישה בלבד (הצגת פסק דין בודד,
או כמה מסמכים נבחרים לחיפוש החכם) — לא כדאונלוד גורף.

בזכות השילוב הזה, הנפח שצריך לחיות ב-Turso הוא כ-2.3-3GB בלבד (לא 5GB) — צפוי
להישאר זמן-מה בתוך המסלול החינמי של Turso (5GB), ולדרוש רק את המסלול הזול ביותר
($4.99/חודש, 9GB) כשהמאגר יגדל מעבר לכך. נבדק ונשלל: מסלולי-חינם של Postgres
מנוהל (Supabase 500MB, Neon 0.5GB) קטנים מדי בסדרי גודל בשביל הצורך הזה, ואירוח
עצמי (VM עם דיסק קבוע) נשלל כי נדרשת פלטפורמה מנוהלת-לגמרי כמו היום.

**שינויים נדרשים (תמצית):**
- `zot/storage.py` — פונקציות חדשות להעלאה/שליפה של טקסט פסק-דין בודד מ-R2
  (`upload_fulltext`/`fetch_fulltext(s)`, עם שליפה מקבילה ל-N מסמכים בבת אחת).
- `zot/ingest.py` — הסרת עמודת `full_text` מטבלת `verdicts`; הפיכת `verdicts_fts`
  לטבלת FTS5 "חסרת-תוכן" (`content=''`) כך שהיא בונה את האינדקס בלי לשמור עותק
  שאילתא-בר של הטקסט המקורי.
- `zot/search.py` — מעבר מ-`sqlite3.connect` מקומי לחיבור Turso; שליפת טקסט מלא
  לפי דרישה בלבד (`get_verdict`, ו-`retrieve_for_ai` עבור מדגם החיפוש החכם).
- `zot/ai_search.py` ו-`zot/config.py` — שלוש תיקונים נפרדים לאיכות החיפוש החכם:
  1. הגדלת `AI_MAX_DOCS` מ-8 ל-20 ומעלה (מודל Sonnet 5 תומך בחלון-הקשר ענק).
  2. הסרת כל אזכור ל"מדגם"/מספר מסמכים מהתשובה המוצגת למשתמש.
  3. זיהוי שאלות מטא כמו "מה ההחלטה הכי מעודכנת במאגר" (שדה `intent` חדש
     בניתוח השאלה) וניתוב ישיר למיון-לפי-תאריך, במקום דירוג BM25 רגיל שלא
     מתאים לשאלות כאלה.
- סקריפט הגירה חד-פעמי + עדכון תהליך הפרסום היומי (`run_daily.bat`) כך שיעלה
  טקסטים בודדים ל-R2 ויריץ `turso db import` במקום להעלות את כל `index.db`.

**סטטוס:** תוכנן ואושר (ראו שיחה זו) — טרם מומש. השלב הראשון שדורש פעולת משתמש
הוא פתיחת חשבון Turso ויצירת מסד נתונים (המשתנים הדרושים, `TURSO_DATABASE_URL`
ו-`TURSO_AUTH_TOKEN`, יתווספו בדומה למשתני ה-R2 הקיימים).
